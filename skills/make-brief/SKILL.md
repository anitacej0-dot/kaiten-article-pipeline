# Навык: make-brief (сборка ТЗ)

Превращает заполненную intake-карточку в нормальное ТЗ, по которому автор напишет
качественный черновик без додумывания. Этап 1 конвейера (см. [CLAUDE.md](../../CLAUDE.md)).

## Вход
- `articles/<папка>/00_intake.md` — заполненная intake-карточка.
- База знаний: `knowledge/seo_policy.md`, `products.md`, `icp_segments.md`, `editorial_policy.md`.
- Шаблон: `templates/article_brief.md`.

## Gate перед стартом
Не собирай бриф, если в intake нет хотя бы одного из: **цель, ЦА, площадка, тип материала,
SEO-вводные**. Вместо брифа верни список недостающего — это вопросы к редактору/SEO.

## Что делать
1. Прочитай `00_intake.md`.
2. Подтяни из базы знаний:
   - **ЦА** → сегмент и роль из `icp_segments.md` (боли, язык, сценарии).
   - **Продуктовая связка** → факты и корректное позиционирование из `products.md`.
   - **SEO** → правила из `seo_policy.md` (мета-теги, ключи, структура под нейросети, Баден-Баден).
   - **Тон и ограничения** → `editorial_policy.md`.
3. Заполни бриф по `templates/article_brief.md`:
   - 2–3 варианта H1 (ключ + конкретная польза);
   - цель текста и как нативно подвести к Kaiten;
   - ЦА с болями (не «вообще»);
   - **интент одной строкой на каждый ключ**;
   - план H2/H3 с тезисами по секциям;
   - обязательные блоки: определение в 2–3 предложения в начале, FAQ, таблицы/списки;
   - **чек-лист техтребований** (мета-теги, URL, перелинковка, text.ru ≥90%);
   - ссылки на конкурентов в выдаче для research.
4. Сохрани в `articles/<папка>/01_brief.md`.
5. Покажи результат и жди подтверждения перед этапом 2 (research).

## Правила
- Только факты из `knowledge/`. Не выдумывай возможности продукта и цифры.
- Соблюдай редполитику и `forbidden_phrases.md` уже в формулировках брифа.
- Бриф — 1–2 страницы: достаточно, чтобы не было структурных ошибок, но автор сохраняет свободу.

## Выход
`01_brief.md` — готовое ТЗ. Следующий этап: `research-article` → `02_sources.md`.
