# Протокол исследования продукта для фактчекинга

## Главное правило

Не полагайся на свои знания о продукте. Данные из обучающей выборки могут быть устаревшими или неточными. Каждый факт должен быть подтвержден через web_search и web_fetch с официального сайта продукта.

## Какие источники разрешены

Разрешены ТОЛЬКО:
- Официальный сайт продукта (главная, раздел возможностей, страница тарифов)
- Официальная база знаний или справочный центр продукта
- **Для самого Kaiten** — официальная база знаний https://faq-ru.kaiten.site/ (плюс kaiten.ru). Продуктовые факты о Kaiten сверяем прежде всего там, а не только по `knowledge/products.md`.
- Официальная документация и API-документация продукта
- Официальный блог компании-разработчика
- Пресс-релизы компании-разработчика на их собственном сайте

## Какие источники ЗАПРЕЩЕНЫ

Запрещены категорически:
- Сайты-отзовики (G2, Capterra, отзовик и т.д.)
- Сторонние обзоры и статьи в других блогах
- Агрегаторы и каталоги ПО
- Форумы и социальные сети
- Видеообзоры на YouTube
- Статьи в СМИ о продукте
- Маркетплейсы приложений (кроме случаев проверки наличия мобильного приложения)
- Любые страницы, которые не принадлежат разработчику проверяемого продукта

Если в поисковой выдаче попадается сторонний источник, ПРОПУСТИ его. Открывай только страницы на домене продукта.

## Пошаговый порядок исследования

### Шаг 1 — Найти официальный сайт

Выполни web_search по названию продукта. Определи домен официального сайта. Все дальнейшие действия выполняй только на этом домене и его поддоменах.

### Шаг 2 — Главная страница

Открой главную страницу через web_fetch. Зафиксируй:
- Позиционирование продукта: для кого, какие задачи решает
- Какие функции выделены как ключевые
- Есть ли отраслевой фокус

### Шаг 3 — Страница возможностей или функций

Найди раздел с возможностями, функциями или продуктовым описанием. Часто это /features, /возможности, /функции или подобный путь. Если не находишь напрямую, выполни web_search: «[название продукта] site:[домен] возможности» или «[название продукта] site:[домен] features».

Зафиксируй:
- Полный список заявленных функций
- Детали работы каждой функции, которая относится к теме подборки
- Какие функции описаны подробно, а какие только упомянуты

### Шаг 4 — Страница тарифов

Найди и открой страницу тарифов. Зафиксируй:
- Названия тарифных планов
- Стоимость каждого тарифа (помесячная и годовая, если есть)
- Наличие бесплатного тарифа или пробного периода
- Ограничения по каждому тарифу: количество пользователей, проектов, объем хранилища
- Какие функции на каких тарифах доступны
- Есть ли индивидуальный или Enterprise тариф

Если есть таблица сравнения тарифов, изучи ее отдельно: там часто содержится информация о функциях, которой нет в других разделах.

### Шаг 5 — База знаний или документация

Найди базу знаний продукта. Для этого выполни web_search:
- «[название продукта] база знаний»
- «[название продукта] справочный центр»
- «[название продукта] help center»
- «[название продукта] документация»

Изучи разделы, которые относятся к теме подборки. Зафиксируй:
- Подробности работы функций, которые упомянуты в проверяемом тексте
- Реальные ограничения, которые не указаны на маркетинговых страницах
- Условия доступа к функциям (на каких тарифах, с какими настройками)
- Интеграции и их детали
- Особенности импорта, экспорта, API

### Шаг 6 — Сводка найденного

После исследования составь внутреннюю сводку (для себя, не для пользователя) по следующим пунктам:
- Какие страницы были изучены (с URL)
- Основные функции, подтвержденные на сайте
- Тарифы и их ограничения
- Функции, релевантные теме подборки, которые нашел на сайте
- Ограничения и особенности, которые заметил

Эта сводка станет основой для сравнения с текстом пользователя.

## Что делать, если информацию не удалось найти

Если на официальном сайте не нашел подтверждения какой-то функции или факта:
- НЕ подтверждай его из памяти
- НЕ ищи в сторонних источниках
- Прямо скажи пользователю: «Информацию о [X] на официальном сайте найти не удалось»
- Порекомендуй пользователю проверить вручную

## Глубина исследования

При фактчекинге не ограничивайся одной-двумя страницами. Минимальный набор для каждого продукта:
- Главная страница — 1 запрос web_fetch
- Страница возможностей — 1-2 запроса web_fetch
- Страница тарифов — 1 запрос web_fetch
- База знаний — 2-3 запроса web_fetch по релевантным разделам

Итого: 5-7 запросов web_fetch на один продукт как минимум. Если продукт крупный и у него много разделов, может потребоваться больше.
