# Workflow: SEO article pipeline

> Единый редакционный цикл для **всех** SEO-статей репозитория. Отлажен на
> `articles/2026-07-chto-takoe-vebinar`. Каждая новая статья идёт по этим этапам и
> stop-points. Общие правила — в [CLAUDE.md](../CLAUDE.md). Статус конкретной статьи
> ведётся в её `pipeline_status.md` (шаблон — `templates/pipeline_status_template.md`).

## Принцип
AI делает первые ~60%, человек отвечает за стратегию на входе и качество на выходе.
**Редактор — финальный владелец качества.** Между этапами есть **stop-points**: без решения
редактора двигаться дальше нельзя.

## Канонические имена файлов
`00_intake` → `01_brief` → `01_brief_review` → `02_research` → `03_outline` → `04_draft` →
`05_editorial_review` → `06_article_score` → `07_revision_task` → `08_revised_draft` →
`09_rescore` → `10_publication_pack` → `11_visual_brief` → `12_visual_assets` →
`13_image_generation_queue`.

> В пилоте про вебинар нумерация местами отличалась (были `04_draft_v2/v3`, `08_draft_v4`) —
> это ad-hoc итерации. Для новых статей используем канонические имена выше.

## Stop-points (сводка)
- **STOP 1 — после `01_brief_review`:** нельзя писать статью, пока бриф не `APPROVED`.
- **STOP 2 — после `03_outline`:** нельзя писать черновик, пока outline не одобрен.
- **STOP 3 — после `04_draft`:** редактор смотрит, не вышел ли текст ИИ-шным, коротким, скомканным или типовым.
- **STOP 4 — после `06_article_score`:** скоринг < 90 → в revision-writer; 90+ → к publication pack.
- **STOP 5 — после `12_visual_assets`:** нельзя генерировать все картинки подряд — сначала `13_image_generation_queue` и минимальный набор; генерация — только по отдельной команде редактора.

## Этапы

### 00_intake.md — Intake
1. **Skill:** — (заполняет редактор по `templates/article_intake_template.md`).
2. **Читает:** ТЗ от SEO/редактора.
3. **Создаёт:** `00_intake.md`.
4. **Готово:** заполнены тема, тип, площадка, цель, ЦА, SEO-вводные, ключи, конкуренты, продуктовая зона, ограничения.
5. **Стоп:** если нет цели / ЦА / площадки / типа / SEO-вводных — не идти дальше, задать вопросы редактору/SEO.
6. **Дальше:** все обязательные поля заполнены → `brief-architect`.
7. **Нельзя пропускать:** пустые обязательные поля; «додумывание» за редактора.

### 01_brief.md — Brief
1. **Skill:** `brief-architect`.
2. **Читает:** `00_intake.md` + `knowledge/` (`seo_policy`, `icp_segments`, `products`, `article_patterns`, `editorial_corrections_patterns`, `weak_article_antipatterns`, `good_examples`, `editorial_policy`).
3. **Создаёт:** `01_brief.md` по `templates/brief_template.md`.
4. **Готово:** подробный бриф — угол, ЦА и боль, интент, формат подачи, продуктовая связка, визуалы, анти-паттерны, критерии качества.
5. **Стоп:** нет критичных вводных → блок «Вопросы перед брифом», не выдумывать.
6. **Дальше:** бриф собран → `brief-reviewer`.
7. **Нельзя пропускать:** типовую структуру без угла; поверхностную боль; рекламную продуктовую вставку.

### 01_brief_review.md — Brief review · 🛑 STOP 1
1. **Skill:** `brief-reviewer`.
2. **Читает:** `01_brief.md`, `00_intake.md`, релевантные `knowledge/`.
3. **Создаёт:** `01_brief_review.md` + статус **APPROVED / NEEDS_REVISION / BLOCKED** (scoring 0–100).
4. **Готово:** статус проставлен, критичные проблемы описаны, есть чёткое «можно/нельзя писать статью».
5. **🛑 STOP 1:** **нельзя писать статью, пока бриф не `APPROVED`.** `NEEDS_REVISION`/`BLOCKED` → вернуть в `brief-architect`.
6. **Дальше:** только при `APPROVED` → `research-article`.
7. **Нельзя пропускать:** APPROVED «для галочки» при слабом угле/боли; продуктовую вставку, выглядящую рекламно.

### 02_research.md — Research
1. **Skill:** `research-article` (+ `product-fact-check` для продуктовых фактов и конкурентов).
2. **Читает:** `01_brief.md`, `knowledge/products`, `icp_segments`, выдачу/конкурентов.
3. **Создаёт:** `02_research.md`.
4. **Готово:** источники (required/optional), конкуренты и их слабые места, вопросы читателя, FAQ, продуктовые факты и границы, внутренние ссылки, вывод «можно ли к outline».
5. **Стоп:** нет источников / структура не отвечает интенту → не идти к outline.
6. **Дальше:** research закрыл интент и продуктовые границы → `build-outline`.
7. **Нельзя пропускать:** выдуманные факты/цифры; неверные продуктовые утверждения (Kaiten — только из `products.md`).

### 03_outline.md — Outline · 🛑 STOP 2
1. **Skill:** `build-outline`.
2. **Читает:** `01_brief.md`, `02_research.md`, `knowledge/seo_policy`, `editorial_policy`, `good_examples`.
3. **Создаёт:** `03_outline.md` (H2/H3 с задачей блока, визуалы, продуктовая связка, риски).
4. **Готово:** структура отвечает интенту, не типовая, обязательные блоки на месте, проверки в конце.
5. **🛑 STOP 2:** **нельзя писать черновик, пока outline не одобрен редактором.**
6. **Дальше:** outline одобрен → черновик (этап 04).
7. **Нельзя пропускать:** механическую схему «что такое → виды → преимущества → как выбрать»; блоки «для объёма».

### 04_draft.md — Draft · 🛑 STOP 3
1. **Skill:** автор (человек/AI) + `kaiten-editorial` (тон/инфостиль); формат подачи — по `article_patterns`.
2. **Читает:** `03_outline.md`, `01_brief.md`, `02_research.md`, `knowledge/editorial_policy`, `forbidden_phrases`, `editorial_corrections_patterns`, `products`.
3. **Создаёт:** `04_draft.md` (+ визуальные пометки `[ВИЗУАЛ: …]`).
4. **Готово:** черновик по outline, угол выдержан, стиль Kaiten, продуктовая связка нативная, self-check в конце.
5. **🛑 STOP 3:** **редактор смотрит черновик** — не вышел ли текст ИИ-шным, коротким, скомканным, типовым; активное действие и ритм на месте.
6. **Дальше:** черновик прошёл беглый взгляд редактора → `05_editorial_review`.
7. **Нельзя пропускать:** нейрослоп и штампы; повторы одной мысли; Kaiten как реклама/платформа не по фактам; воду вместо конкретики.

### 05_editorial_review.md — Editorial review
1. **Skill:** `ai-pre-review` (рубрика + Vale + русские сервисы) + редактор.
2. **Читает:** `04_draft.md`, `02_research.md`, `knowledge/` (`editorial_policy`, `forbidden_phrases`, `bad_patterns`, `products`, `good_examples`), `styles/Kaiten/`.
3. **Создаёт:** `05_editorial_review.md` (замечания по направлениям, HARD/SOFT, без переписывания).
4. **Готово:** разобраны логика, стиль, глубина, повторы, активное действие, ритм, продуктовая связка, форматирование, визуальные пометки, слабые места.
5. **Стоп:** — (ревью — вход в скоринг).
6. **Дальше:** → `06_article_score`.
7. **Нельзя пропускать:** молчаливую правку вместо замечаний; пропуск HARD-нарушений редполитики; выдуманные продуктовые факты.

### 06_article_score.md — Article score · 🛑 STOP 4
1. **Skill:** `ai-pre-review` (скоринг 100).
2. **Читает:** `04_draft.md`, `05_editorial_review.md`, `01_brief.md`, `02_research.md`.
3. **Создаёт:** `06_article_score.md` (оценка по критериям, итог 0–100, решение).
4. **Готово:** проставлен балл и решение; при < 90 — список правок для revision-writer.
5. **🛑 STOP 4:** **< 90 → идти в revision-writer** (`07_revision_task` → `08_revised_draft`). **90+ → к publication pack** (этап 10).
6. **Дальше:** по правилу выше.
7. **Нельзя пропускать:** APPROVED к публикации при < 90; рекламную/сомнительную продуктовую связку.

### 07_revision_task.md — Revision task
1. **Skill:** редактор/`brief-reviewer`-логика собирает ТЗ (из `05` и `06`).
2. **Читает:** `05_editorial_review.md`, `06_article_score.md`, `04_draft.md`.
3. **Создаёт:** `07_revision_task.md` (что / почему / где / **что не трогать**).
4. **Готово:** точечное ТЗ, приоритеты 🔴/🟡, зафиксировано, что нельзя менять (угол, структура, полезные блоки).
5. **Стоп:** —.
6. **Дальше:** → `08_revised_draft`.
7. **Нельзя пропускать:** правки, ломающие угол/структуру; расширение объёма водой.

### 08_revised_draft.md — Revised draft
1. **Skill:** автор (revision-writer) + `kaiten-editorial`.
2. **Читает:** `07_revision_task.md`, `04_draft.md`.
3. **Создаёт:** `08_revised_draft.md` (точечные правки, не переписывание с нуля).
4. **Готово:** все правки из ТЗ внесены, угол/структура сохранены, self-check в конце.
5. **Стоп:** —.
6. **Дальше:** → `09_rescore`.
7. **Нельзя пропускать:** новые смысловые блоки без запроса; смену угла; воду вместо конкретики.

### 09_rescore.md — Rescore
1. **Skill:** `ai-pre-review` (повторный лёгкий скоринг).
2. **Читает:** `08_revised_draft.md`, `07_revision_task.md`, `06_article_score.md`.
3. **Создаёт:** `09_rescore.md` (проверка закрытия правок + новый балл + решение).
4. **Готово:** правки подтверждены, балл обновлён.
5. **🛑 (возврат к STOP 4):** если всё ещё < 90 — снова revision-writer; 90+ → publication pack.
6. **Дальше:** 90+ → `10_publication_pack`.
7. **Нельзя пропускать:** незакрытые 🔴-замечания; вернувшуюся типовую подачу.

### 10_publication_pack.md — Publication pack
1. **Skill:** `package-for-cms`.
2. **Читает:** финальный черновик (`08_revised_draft`), `knowledge/seo_policy`, `internal-links-map`, `target-keywords`.
3. **Создаёт:** `10_publication_pack.md` (H1-варианты, Title ≤70, Description ≤160, slug, лид, теги, внутренние ссылки, CTA, visual brief-рекомендации, чек-лист публикации).
4. **Готово:** пакет собран; отмечено, есть ли визуальные рекомендации.
5. **Стоп:** — (создаётся **только** после rescore 90+).
6. **Дальше:** есть визуальные рекомендации → `visual-producer-kaiten` (этап 11).
7. **Нельзя пропускать:** переспам ключей (Баден-Баден); Title/Description сверх лимитов; неготовый SEO-пакет.

### 11_visual_brief.md — Visual brief
1. **Skill:** `visual-producer-kaiten` (`.claude/skills/`).
2. **Читает:** финальный черновик, `10_publication_pack.md`, `.claude/skills/design-system-kaiten-v01/SKILL.md`.
3. **Создаёт:** `11_visual_brief.md` по `templates/visual_brief_template.md`.
4. **Готово:** определено, какие визуалы нужны и какие — нет; ограничения дизайн-системы; что отдать дизайнеру.
5. **Стоп:** —.
6. **Дальше:** → `12_visual_assets`.
7. **Нельзя пропускать:** картинки «для красоты»; выдуманный интерфейс продукта (нужен реальный скриншот); противоречие дизайн-системе.

### 12_visual_assets.md — Visual assets · 🛑 STOP 5
1. **Skill:** `visual-producer-kaiten`.
2. **Читает:** `11_visual_brief.md`, дизайн-систему.
3. **Создаёт:** `12_visual_assets.md` (обложка, схемы, иллюстрации, таблицы, скриншоты, промпты, alt, подписи, статусы).
4. **Готово:** у каждого визуала — тип, место, зачем, статус.
5. **🛑 STOP 5:** **нельзя генерировать все картинки подряд.** Сначала `13_image_generation_queue` и минимальный набор.
6. **Дальше:** → `13_image_generation_queue`.
7. **Нельзя пропускать:** стоки/мемы/визуальный шум; много цветов вместо одного акцента.

### 13_image_generation_queue.md — Image generation queue
1. **Skill:** `visual-producer-kaiten`.
2. **Читает:** `11_visual_brief.md`, `12_visual_assets.md`, дизайн-систему.
3. **Создаёт:** `13_image_generation_queue.md` — минимальный набор в работу: production-ready промпты, ТЗ дизайнеру на схемы, чек-лист скриншотов.
4. **Готово:** очередь готова; выбран минимум визуалов.
5. **🛑 Стоп:** **картинки генерируются только из этого файла и только по отдельной команде редактора.**
6. **Дальше:** генерация выбранных визуалов (по команде) → доводка дизайнером → вставка в статью → публикация.
7. **Нельзя пропускать:** генерацию «всего сразу»; текст на изображении/нечитаемые надписи; выдуманный интерфейс вместо реального скриншота.

---

## Кто проверяет на stop-points
- STOP 1 — редактор (по `brief-reviewer`).
- STOP 2 — редактор.
- STOP 3 — редактор (беглый взгляд на черновик).
- STOP 4 — по скорингу (`ai-pre-review`) + редактор.
- STOP 5 — редактор (команда на генерацию).

## Сквозные навыки
`kaiten-editorial` (тон/инфостиль на любом тексте), `product-fact-check` (факты о продуктах в подборках), `make-cover` (обложка по фирменному шаблону), `product-roundup-v2` (формат подборки «под ключ»).
