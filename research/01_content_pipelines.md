# Research: открытые решения для контент-конвейеров

**Задача:** найти и сравнить open-source решения, где с помощью AI организован процесс
создания статей: intake → бриф/ТЗ → research → структура → черновик → AI-проверка
качества → редактура. Для каждого — что это, ссылка, какие этапы покрывает, какие
проверки качества использует и что применимо к нашим узким местам.

**Наши узкие места (из карты текущего процесса):**
1. **Ручная редактура** — редактор тратит 1–3 часа чистого времени, правит логику и структуру, а не только язык. Главное бутылочное горлышко.
2. **Нет research-этапа** — непонятно, какие источники смотрел автор, проверял ли конкурентов и продуктовые факты.
3. **Слабые ТЗ** — плохие вводные → черновик с плохой логикой.

> Дата research: июнь 2026. Все ссылки и факты (звёзды, лицензии) проверены по GitHub/сайтам проектов на эту дату.

## Категория A. Research-движки (закрывают отсутствие research-этапа)

### 1. STORM / Co-STORM (Stanford OVAL)
- **Ссылка:** https://github.com/stanford-oval/storm
- **Что это:** система от Стэнфорда, которая исследует тему в интернете и пишет статью с цитатами «с нуля» (изначально — статьи в стиле Википедии). Подкреплена научными статьями (NAACL 2024, EMNLP 2024). ~29k ⭐, MIT.
- **Какие этапы покрывает:** четыре модуля — (1) **Knowledge Curation** (сбор знаний через симуляцию диалогов «автор ↔ эксперт» с разных перспектив), (2) **Outline Generation** (иерархическая структура), (3) **Article Generation** (наполнение структуры с цитатами), (4) **Article Polishing** (полировка, удаление дублей).
- **Проверки качества:** всё привязано к источникам (You.com, Bing, Tavily, DuckDuckGo, свои документы). Ключевая фишка — **«perspective-guided question asking»**: система задаёт вопросы с разных точек зрения, как разные читатели. Co-STORM добавляет **human-in-the-loop**: модератор + эксперты + человек, который рулит диалогом.
- **Что берём нам:** **модель research-этапа.** Их «вопросы с разных перспектив» = наш пункт «вопросы читателя» в research-файле. Можно воспроизвести как skill, который перед структурой генерит карту источников + список вопросов, на которые статья обязана ответить. Закрывает **узкое место №2**.

### 2. GPT Researcher
- **Ссылка:** https://github.com/assafelovic/gpt-researcher
- **Что это:** автономный агент глубокого research по теме (веб + локальные документы). ~28k ⭐, Apache 2.0.
- **Какие этапы покрывает:** только research (но глубоко). Архитектура **planner + execution**: планировщик генерит исследовательские вопросы, исполнители собирают данные, publisher агрегирует в отчёт. Есть **Deep Research** — рекурсивный обход подтем «вширь и вглубь».
- **Проверки качества:** снижение галлюцинаций через объём — агрегирует 20+ источников и берёт наиболее частую информацию; инлайн-цитаты. Честно пишут: «не убираем предвзятость, а максимально снижаем».
- **Что берём нам:** готовый паттерн **research как отдельный модуль с выходным артефактом** (`02_sources.md`). Можно подключить как MCP-сервер прямо в Claude Code. Закрывает **узкое место №2**.

## Категория B. Полные мультиагентные конвейеры

### 3. ALwrity
- **Ссылка:** https://github.com/AJaySi/ALwrity
- **Что это:** AI-платформа для диджитал-маркетинга. MIT, помечена WIP, но рабочие модули есть.
- **Этапы:** Blog Writer идёт по цепочке **Research → Outline → Content → SEO → Publish** с «guarded navigation» (нельзя перескочить этап). Структура генерится до написания.
- **Проверки:** делает ставку на **grounding** — поисковые провайдеры (Exa, Tavily, Serper) с автофоллбэком, интеграция с Google Search Console как источник фактов, цитаты «уменьшают галлюцинации и переписывания».
- **Что берём нам:** идея **«guarded navigation»** = наши quality gates (нельзя идти дальше без выполненного этапа). Подтверждает порядок: research и структура — ДО черновика. Релевантно для SEO-пакета.

### 4. Content-Generation-Workflow (на CrewAI)
- **Ссылка:** https://github.com/Bhavik-Jikadara/Content-Generation-Workflow
- **Что это:** пример мультиагентной команды на CrewAI. MIT, Streamlit-UI.
- **Этапы:** **Research → Planning → Writing → Image → SEO → Editorial Review** — каждый агент со своей ролью; редактура стоит финальным этапом.
- **Проверки:** «editorial checks» как финальный gate (детали правил не расписаны).
- **Что берём нам:** наглядная **карта ролей** (research/план/текст/SEO/редактура как отдельные «должности»-агенты). Подтверждает наш подход «этап = роль с входом и выходом». Сам CrewAI нам, вероятно, не нужен (мы на Claude Code skills), но раскладка ролей — полезный образец.
- **Похожие:** [crewAI-examples](https://github.com/crewAIInc/crewAI-examples) (Content Creator Flow), [awesome-crewai](https://github.com/crewAIInc/awesome-crewai) — каталог community-проектов.

## Категория C. Claude Code / Agent-Skill решения (наш стек — берём в первую очередь)

> Эти два решения построены на том же подходе, что и наш конвейер лендингов (skills + knowledge-файлы + slash-команды), поэтому переносятся к нам почти напрямую.

### 5. SEOMachine ⭐ самый близкий к нам
- **Ссылка:** https://github.com/TheCraigHewitt/seomachine
- **Что это:** **готовый Claude Code workspace** для длинных SEO-статей. MIT, ~7k ⭐.
- **Этапы (как slash-команды):** `/research` (ключи, конкуренты, гэпы) → `/write` (статья 2000–3000+ слов) → `/optimize` (SEO-аудит, готовность к публикации) → `/publish-draft` (WordPress API). Отдельная ветка обновления старого: `/analyze-existing` (health score 0–100) → `/rewrite`.
- **Структура workspace — прямой образец для нас:**
  - **Knowledge-файлы:** `brand-voice.md`, `style-guide.md`, `seo-guidelines.md`, `writing-examples.md` (3–5 эталонных постов), `target-keywords.md`, `internal-links-map.md`, `competitor-analysis.md`. **Это почти один-в-один наш план `knowledge/`.**
  - **26 skills** по категориям + **16+ slash-команд** на каждый этап.
  - **10 агентов**, в т.ч. `Editor` (делает текст «человеческим», выдаёт **humanity score 0–100**), `Content Analyzer` (интент, плотность ключей, readability).
- **Проверки качества (конкретные пороги):** мин. 2000 слов; плотность ключа 1–2%; ключ в H1 и первых 100 словах; 3–5 внутренних ссылок; иерархия H1>H2>H3; уровень чтения «8–10 класс»; длина предложения 15–20 слов; абзацы 2–4 предложения; набор метрик-скоров 0–100.
- **Что берём нам:** **это эталон организации репозитория конвейера.** Прямо переносятся: раскладка `knowledge/`, команды по этапам, идея числовых порогов и **humanity/health score** как автоматического сигнала качества → закрывает **узкое место №1 (ручная редактура)** и **№3 (слабые ТЗ, т.к. бриф собирается из knowledge-файлов)**.

### 6. Cody Article Writer
- **Ссылка:** https://github.com/ibuildwith-ai/cody-article-writer
- **Что это:** **Agent Skill** (совместим с Claude Code / Claude / Copilot / Codex). Custom-лицензия: бесплатное использование для создания контента, но без редистрибуции кода.
- **Этапы (10 фаз):** идея → (опц.) план research → выбор стиля → заголовок/тезис → структура → написание с инлайн-цитатами → **approval** → (опц.) проход редактора → метаданные (frontmatter) → экспорт.
- **Проверки качества:**
  - **Approved sources** — пользователь утверждает источники до того, как AI их использует.
  - **Required / optional** теги на источники (must use / use if relevant).
  - **Reusable style guides** — переиспользуемые голос/тон/глубина, применяются по фазам.
  - **Итеративные gate'ы:** «ничего не идёт дальше, пока ты не доволен» — каждая фаза заканчивается явным подтверждением.
- **Что берём нам:** механика **approved sources + required/optional** напрямую закрывает «непонятно, какие источники смотрел автор» (**узкое место №2**). **Reusable style guides** = переиспользуемая редполитика на входе (**узкое место №3**). Фазовые approval-gate'ы = наши quality gates.

## Категория D. Проверки качества как код (редполитика → авто-проверки)

### 7. Vale — линтер прозы ⭐ ключевая находка для редактуры
- **Ссылка:** https://github.com/vale-cli/vale · сайт https://vale.sh
- **Что это:** open-source линтер прозы (как линтер кода, но для текста). MIT, индустриальный стандарт в технической документации (Elastic, Microsoft, Google style guides импортируются готовыми).
- **Как работает:** ты **«кодифицируешь» редполитику** в правила. Vale проверяет текст и выдаёт **errors / warnings / suggestions** — это **ровно наша модель HARD (блокирует) / SOFT (предупреждает)** из конвейера лендингов. Работает офлайн (текст никуда не уходит), знает Markdown/HTML/AsciiDoc.
- **Что берём нам:** **это и есть автоматизация редполитики.** Запреты, штампы, англицизмы, «нейрослоп», требования к лидам — оформляются как Vale-правила и гоняются на каждом черновике ДО редактора. Снимает с редактора рутину «исправлять формулировки» → закрывает **узкое место №1**. Один из самых конкретных и переносимых результатов research.

## Лучшие практики — что реально стоит взять нам

Сведено к тому, что бьёт по нашим трём узким местам.

### Против отсутствия research-этапа (№2)
1. **Сделать research отдельным обязательным этапом с артефактом** (`02_sources.md`) — как в GPT Researcher и STORM. Без файла источников черновик не запускается (quality gate).
2. **Перспективные вопросы читателя** (из STORM): research-skill генерит список вопросов, на которые статья обязана ответить, до написания структуры.
3. **Approved sources + required/optional** (из Cody): автор фиксирует и помечает источники; видно, что реально изучено, проверены ли продуктовые факты и конкуренты.

### Против слабых ТЗ (№3)
4. **Бриф собирается автоматически из knowledge-файлов** (из SEOMachine): `brand-voice.md`, `style-guide.md`, `seo-guidelines.md`, `writing-examples.md`, ICP. Это наш план `knowledge/` — SEOMachine подтверждает раскладку почти один-в-один.
5. **Reusable style guides** (из Cody): редполитика и тон подаются на входе как переиспользуемый объект, а не пересказываются каждый раз.
6. **«Guarded navigation» / quality gates** (из ALwrity): нельзя перейти к черновику без брифа, research и структуры.

### Против ручной редактуры (№1) — главный приоритет
7. **Vale: редполитика как код.** Штампы, англицизмы, нейрослоп, требования к лидам → HARD/SOFT-правила, прогон на каждом черновике до редактора. Снимает с редактора рутинную чистку языка.
8. **AI-предредактура перед редактором + self-QA loop:** AI сам критикует черновик и возвращает автору, пока не исправлены high-критичные замечания. Редактор получает уже очищенную версию.
9. **Числовые scoring-пороги** (из SEOMachine: health/humanity/readability 0–100): объективный сигнал «готово к редактору / вернуть автору» вместо ручной оценки на глаз.
10. **Редактор остаётся финальным владельцем качества** (human-in-the-loop, как Co-STORM): автоматизация чистит вход, человек принимает финальное решение. Совпадает с принципом из нашей карты процесса.

### Вывод по стеку
Нам ближе всего **Категория C** (SEOMachine, Cody) — тот же стек Claude Code + skills + knowledge, что и в конвейере лендингов, поэтому переносится почти напрямую. **Vale** берём как отдельный инструмент проверки редполитики. **STORM / GPT Researcher** — как образец (или прямой модуль через MCP) для research-этапа. **CrewAI-примеры и ALwrity** — как референс по раскладке ролей и порядку этапов, но не как основа.
